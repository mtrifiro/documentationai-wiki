---
title: Almost-Right
description: On the class of bugs that only exist under conditions you haven't tested yet.
---
There are three kinds of code. Wrong code, almost-right code, and actually-right code. The first kind does not compile, fails the obvious test, gets caught at the moment of writing, and never makes it to production. The third kind handles every case correctly: the concurrent calls, the partial failures, the retries, the orderings, the recoveries. It runs cleanly under conditions the writer anticipated and conditions the writer did not. The first kind is cheap. The third kind is expensive.

The middle kind — almost-right code — is code that compiles. It passes the unit tests. It looks correct in the diff. It handles the cases the requirement described. A reviewer reading it sees nothing obviously wrong. The continuous-integration pipeline turns green. The deploy goes out. The first thousand requests work exactly as expected. The system runs for a week, or a month, or six weeks, and nobody notices anything. Then a customer reports a charge that ran twice. An accountant finds a $40 discrepancy in the daily reconciliation. The on-call engineer wakes up at 3 am to a batch job that crashed because the upstream queue was empty for the first time in eleven months. The almost-right code, which had been almost-right the entire time, has finally encountered the specific condition it was almost-right about, and the system has broken in a small and specific way.

The defining feature of almost-right code is that the bug it contains is not visible at the moment of writing. The writer does not see it. The reviewer does not see it. The test suite does not see it. The CI does not see it. The deploy does not see it. The first ten thousand executions of the code do not see it. The bug lives in the conditions the code did not encounter during any of these checks: concurrent access from a second request, a network blip that triggered a retry, a downstream service that returned a malformed response, and a process that died after the database commit but before the response was sent. Each of these conditions is rare on any single request and certain across enough requests. The almost-right code handles every case the writer thought to test and fails the cases that show up later as production conditions accumulate. The bug is real. The bug was real the day the code shipped. The bug is detected weeks later by the conditions, not by the code.

This is not a new class of bug. Distributed-systems engineers have been writing about it for forty years. What is new is the rate at which the class is being introduced. For most of computing history, the bug class was bounded by the rate at which careful humans wrote code, and careful humans, after enough years on the job, developed instincts that caught a meaningful fraction of almost-right code before it shipped. The senior engineer would read the diff, ask *what happens if this runs twice*, and the writer would add the idempotency check. The architecture review would catch the race condition before the design committed. The team's collective scar tissue, built up over a decade of incidents, served as a leaky but functional filter against almost-right code reaching production. The filter was expensive because it required hiring and retaining the kind of engineers who had been burned enough times to be careful, but it worked at the volume of code humans could write.

The filter does not work at the volume of code agents write. The senior engineer cannot read a thousand lines an hour and catch the race condition. The agents do not have the senior engineer's instincts. The architecture review cannot scale to twenty deploys a day. The team's scar tissue, which used to be the difference between *might be almost right* and *actually right*, is no longer in the path. The agents write almost-right code at a rate the team cannot review at the depth required, and the almost-right code ships, and the rate of production incidents climbs in proportion. The shape of the problem is identical to the shape it has always had. The volume is new.

Why do agents produce almost-right code rather than actually-right code or outright wrong code? Three reasons. First, agents are trained on a corpus that overwhelmingly contains almost-right code, because most production code is almost right, including the code that worked for a year before breaking, which is the version that ends up in the training set. The pattern the agent learns is the pattern of code that mostly works. Second, agents reason locally. They see the function, not the system. The conditions that turn almost-right into broken are global properties of the system: what other code touches this row, what client retry behavior exists, and what queue ordering is in play. The agent has no view of them. Third, agents have no consequence model. They have not been paged. They do not know what 3am production looks like. The defensive reflexes that careful humans developed because of consequences are not built into the agent's reasoning. The agent writes the obvious version, the obvious version is almost right, and the agent considers the job done.

The standard response to almost-right code is to harden detection. Better static analysis. Smarter linters. AI reviewers that flag dangerous patterns. More tests. Property-based testing. Chaos engineering. Each of these catches some additional fraction of almost-right code before it ships. Each of these has a ceiling because the defining property of almost-right code is that the bug is not visible at the moment of writing. That means the detector, by reading the code at that same moment, has the same blind spot the writer had. The detector can catch some patterns. It cannot catch the cases where the bug is in the interaction between the code and conditions that do not exist yet. The detection strategy plateaus, and the plateau is below the level at which the problem is solved.

Reboot's response is different. Stop trying to detect almost-right code. Make the conditions under which almost-right code fails unreachable. The race condition is almost-right until two requests hit the same row. Reboot serializes per-entity access; two requests cannot hit the same row simultaneously. The race condition does not turn into a bug because the condition that would have made it a bug does not occur. The retry duplication is almost-right until a retry fires twice. Reboot executes actions exactly once across crashes; the retry cannot fire twice. The duplicate does not happen because the condition does not occur. The almost-right code stays almost right. The system stays correct because the runtime has removed the conditions that turn almost right into broken.

This is the structural shift. Almost-right code is not a problem solved by making the code more right. It is a problem solved by making the conditions that expose its almost-rightness unreachable. The runtime owns the conditions. The application stays almost right. The system behaves like the code was actually right, because the runtime made the gap invisible from the outside.

That is the almost-right problem. Reboot is the runtime that closes it, not by making bad code good, but by making the world the bad code runs in safe.
