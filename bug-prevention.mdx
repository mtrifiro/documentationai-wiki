---
title: Bug-Prevention
description: 'How the runtime closes bug categories by construction, not by review.'
---
Here is a charge that ran twice. The customer hit Pay. The browser spun. The network blinked. The browser retried. Both requests reached the server. Both loaded the cart, both called the payment processor, both wrote a Charge row to the database with the same idempotency key — except the agent that wrote the endpoint generated the idempotency key from `uuid4()` inside the handler, so the two requests had two different keys. Stripe processed both. The customer was charged $89.40 twice. The Charge rows are sitting in the database, both marked `succeeded`, both with their own `stripe_charge_id`, looking like two completely legitimate purchases. Nothing in the code is wrong, exactly. Every line does what it says. The bug lives in the seam between two correct executions.

That is the shape of the bugs Reboot prevents. They are not syntax errors. They are not logic errors that show up in unit tests. They are correctness failures that happen at the seams — between two requests, between a process and its successor, between a worker and the queue, between the database commit and the network response. The code looks fine in review. The bug is invisible until production conditions stress the seam.

Take a lost write. An agent writes a function that reads a user's `balance`, subtracts an amount, and writes it back. Two transfers arrive concurrently. Both read $1,000. Both write $900. The customer thinks they sent two transfers of $100; the balance went down by $100. The missing $100 isn't logged anywhere as missing — both writes succeeded. The audit log shows two successful transfers. The ledger shows one. The discrepancy is found three weeks later by an accountant. The code review never had a chance — the read-modify-write was three lines and looked exactly like every other Django view in the codebase. In Reboot, that function lives inside an entity action. The runtime serializes access to the entity. The second read sees $900, not $1,000. The write produces $800. The bug is unreachable because the runtime refuses to interleave the two actions.

Take a stale read. A user hits Save on their profile. The form posts to the server. The server writes the new name to the database, commits, and returns 200. The user's browser redirects to the profile page, which loads the profile from a read replica that hasn't caught up yet. The user sees their old name and refreshes — now it's correct. They assume the save didn't work the first time and don't trust the page. Multiply this across a customer service flow and you get tickets. The agent writing the page had no way to know the read replica lagged. The reads-after-writes consistency model lived in the infrastructure docs, not the application logic. Reboot's runtime gives the action read-your-writes by construction — the next read inside the same logical chain returns the value that was just written, every time. The seam between write and read disappears because the runtime owns both.

Take a duplicate event. A webhook from Stripe fires when a charge succeeds. The handler creates an invoice, sends an email, marks the charge `processed`. The handler crashes after sending the email but before marking the charge — say a Kubernetes pod gets evicted mid-handler. Stripe doesn't see a 200 and retries the webhook. The new handler runs, creates a second invoice, sends a second email, finally marks `processed`. The customer has two invoices and two emails. The agent wrote the handler as straight-line code. There was nowhere obvious in the code to ask "what if I run twice." Reboot's actions execute exactly once across crashes — the runtime journals each step, picks up where the crashed action left off, and finishes. The second webhook arrives, the runtime sees the action is already complete, the duplicate is a no-op. The bug never reaches the customer.

Take a negative inventory count. An e-commerce site has 5 units of an item left. A flash sale starts. 200 users hit Add to Cart at once. 200 endpoints each read inventory = 5, each subtract 1, each write inventory = 4. The system sells 200 units. Five units exist. 195 customers get an order confirmation for a product that will never ship. The operator finds out the next morning. The agent who wrote the endpoint did the obvious thing. The runtime in front of the entity makes the obvious thing impossible — only one decrement runs at a time against the item, the second decrement reads the post-decrement value, and after five decrements the entity rejects further reservations. The system sells 5 units. 195 customers get a "sold out" message instead of a refund queue.

Take a 3am batch crash. A nightly job aggregates the day's transactions into a summary table. For six months it has run cleanly. One night the upstream feed is empty — a public holiday in a country that publishes the feed. The agent's code does `feed[0].timestamp` to anchor the batch. `feed` is an empty list. The job crashes. The summary doesn't run. Three downstream dashboards show stale data the next morning, and a finance team makes a decision against a stale number. The agent never wrote a test for an empty feed because it had never seen one. The bug is not in the aggregation logic — it's in the assumption that the upstream is always populated. In Reboot, the action handles the empty case by construction: the action either ran successfully (with whatever rows it had, including zero) or it failed and was retried with exponential backoff, never silently corrupting the downstream state. The runtime separates "ran with empty input" from "crashed mid-run" — the agent doesn't have to think about which.

Take an out-of-order delivery. A service publishes events. A downstream service consumes them. The agent wires the producer with a fire-and-forget call. Most events arrive in order. Some don't — a network blip retries an event after a later event already landed. The consumer processes them in arrival order, which is wrong for state machines (you can't process a `cancel` before the corresponding `create`). Bugs appear weeks later as ghost records. The agent had no way to know the queue's ordering guarantee, because the queue's ordering guarantee was somewhere on page 12 of the SQS docs. Reboot's actions emit events with per-entity ordering guaranteed by the runtime; the consumer sees them in the order they were emitted, even after retries.

The bugs share a pattern. Each one is a place where the code is locally correct and the seam between executions is what breaks. Each one is invisible to a reviewer who reads the code linearly. Each one is the kind of bug the agent has no instinct for, because the agent has not been paged for it at 3am. And each one is the kind of bug the runtime can make structurally impossible — not by hoping the writer remembers to handle it, but by owning the seam where it lives.

This is what bug prevention looks like when you push it into the substrate. Not better tests. Not better review. Not better prompting. The bugs are gone because the states where they exist are not states the runtime allows. The duplicate charge does not happen. The lost write does not happen. The 3am crash does not happen. The agent writes naive code. The system runs correct. The bugs that used to fill the postmortem queue are not bugs anymore — they are categories the runtime closed by construction.
