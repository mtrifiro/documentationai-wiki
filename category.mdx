The first question anyone asks about Reboot is _what is it like?_ People reach for the nearest category and try to slot it in. _Is it a framework? Is it a workflow engine? Is it a BaaS (Backend-as-a-Service)? Is it Temporal-but-different? Is it Erlang-but-with-types? Is it Firebase for agents?_ Every comparison is close enough to be plausible and wrong enough to mislead. The reason every comparison fails is the same — Reboot is not a better version of an existing category. It is a category that didn't exist before because the conditions that produce it didn't exist before.

## Categories Form When the Inputs Change

Categories in infrastructure form when the inputs change. Containers became a category when virtualization got cheap enough that "ship a process with its environment" stopped being a science project. The relational database became a category when storage got cheap enough that normal forms stopped being theoretical and started being practical. The browser became a category when the network got fast enough that documents could ship across it in real time. In each case, the category did not exist, then the underlying conditions shifted, then the category appeared — and within a decade nobody could remember what it was like to build software without it.

## The Shifted Condition

The condition that has shifted now is who writes the code. For sixty years, the answer was: a careful human. Every framework, every library, every runtime, every language design choice took that answer as ground truth. The careful human was the unit of trust. The careful human held the locks. The careful human read the docs. The careful human knew when to reach for the escape hatch and when not to. The entire stack — from C's pointer arithmetic to Kubernetes' YAML — is calibrated for somebody who knows what they're doing and what they're not doing. Power tools assume a craftsman. The whole system assumes the craftsman.

The careful human is no longer the unit of code production. The agent is. The agent writes fast, writes plausibly, writes code that compiles and passes the obvious tests and looks indistinguishable from code a junior engineer would have written. What the agent does not have is the craftsman's scar tissue — the years of having gotten distributed code wrong in ways that produced real pages at real 3ams. The agent has no model of which lines are load-bearing and which lines are decorative. It writes both with the same confidence.

## Why Existing Comparisons Fail

Every existing infrastructure category was built for the craftsman. Calling Reboot a _framework_ misses this. Frameworks are tools the craftsman opts into and configures. They have surfaces — config files, lifecycle hooks, middleware chains — that assume somebody who knows what each surface does. Agents do not know what each surface does. Agents will reach into the lifecycle hook and write something that compiles and breaks under load, and the framework will obediently run it. The framework's contract with the craftsman — _I give you power, you use it responsibly_ — does not survive contact with an actor who will use the power confidently and irresponsibly.

Calling Reboot a _workflow engine_ misses it in the other direction. Workflow engines like Temporal and Step Functions take ownership of durability and retries, which is closer to the right move. But they are bolt-on. The application sits beside the workflow engine and calls into it. The application can still hold its own state. The application can still bypass the workflow engine when the developer thinks it's "just a small thing." Workflow engines are libraries dressed as platforms. The agent will dress them back down — wire them in halfheartedly, store state in the wrong place, and write the durable parts of the workflow as undurable side effects because the durable surface was "too much ceremony for this case." The bolt-on contract — _use me if you remember to_ — does not survive an actor who doesn't know when to remember.

Calling Reboot a _backend-as-a-service_ misses it on the surface layer. BaaS products like Firebase or Supabase take ownership of the database and the auth flow. They do not take ownership of the application's logic. The logic lives in cloud functions, which are stateless, isolated, and each individually exposed to every distributed-systems hazard the application has ever faced. The hard parts — the parts where two cloud functions race, or one fails midway, or the same event triggers two of them — are not in scope. BaaS gives you a hosted database. The trust gap moves up one layer and is exactly as wide as it was.

## A Different Premise

Reboot's category is none of these because Reboot's category has a different premise. The premise is: the code is being written by an actor that does not understand the distributed-systems implications of what it writes, and the runtime is going to be safe anyway. That premise has never been the design center of an infrastructure product before. Frameworks assumed the developer would learn the framework. Workflow engines assumed the developer would learn the workflow primitives. BaaS assumed the developer would learn the API. Every previous category required the actor on the other end to be teachable. Reboot does not require the agent to be teachable, because the surface the agent writes against does not contain the things that need teaching. The dangerous decisions are not exposed. The agent cannot make them, correctly or incorrectly. The runtime made them once, for everyone, at the layer below.

## Category, Not Feature

This is what makes it a category and not a feature. A feature is something you can add to an existing product. _Temporal could add this. Firebase could add this. Kubernetes could add this._ No, they cannot — not without breaking the contract that defines them. Temporal's contract is _you write workflows in code; we make them durable_. The "you write workflows in code" part is the surface that needs to disappear in the new category. Firebase's contract is _you write cloud functions; we host them_. The "you write cloud functions" part is the surface that needs to disappear. Each existing category is defined by a surface area that the new category eliminates. You cannot eliminate the defining surface of a category and call it the same category. The result is something else. Something new.

Categories matter because they organize the future. Once a category exists and has a name, every conversation about the space gets faster. _Which database are we using? Which orchestrator? Which BaaS?_ The question presupposes the category. The category presupposes a design center. The design center constrains what can be built. For the last decade, the design center for infrastructure has been _make the craftsman more productive_. Reboot's design center is _make the craftsman unnecessary for the parts that are hard_. That is not the same product, sold differently. It is a different product because the design center is different because the inputs to the system are different.

## The Category Is Already Forming

Naming the category honestly is the unlock. Stop trying to make it a faster framework or a smarter workflow engine or a more opinionated BaaS. It isn't. It is the runtime for the era when most code is written by AI. The conditions that produce that category are already here. The category is already forming. Reboot is what it looks like when somebody builds for it on purpose.
