---
title: Fewer Questions
description: Don't train better answers. Remove the questions.
---
Watch an AI agent write a backend endpoint. It declares the route. It parses the input. It opens a database connection. Somewhere around line twelve it has to make a decision, and the decision is invisible — to the agent, to the reviewer, to the test suite. The decision is: *what isolation level am I assuming this transaction runs at?* The agent doesn't know it made the decision. The code reads, computes, writes, and returns. It looks like one operation. It is, in the runtime, four operations interleaved with operations from every other request happening at the same moment. Whether the four operations cohere into a correct result depends on the answer to a question the agent never knew it was answering.

Scale this out across a service. Every endpoint is a small bundle of unanswered questions. *Is this idempotent? What happens if the client retries? What if the process dies between the write and the response? What if another request gets to this row a microsecond earlier? What if the downstream call times out — do I retry, fail forward, compensate, or wait? What ordering guarantee does this queue actually give me?* Each question has a right answer that depends on the specific business logic, the specific load profile, the specific failure modes the system has to survive. Each question has a thousand wrong answers that look exactly like the right one in code review.

A senior engineer reading the same code reads the questions as they pass. They don't always answer them correctly either, but they *see* them. The instinct is built from a decade of having gotten the answers wrong in production. The agent has no such instinct. The agent reads its own training data, which is full of code that mostly worked because the engineer who wrote it had already answered the questions, often implicitly, often in a comment that didn't make it into the snippet. The agent inherits the answer and not the question. It writes the function the way the function looked in the training set, and it ships, and the question goes unanswered, and the production behavior is whatever the default happens to be.

The reflex response is to teach the agent to ask the questions. Prompt the agent with the right checklist. Train it on better examples. Build evaluators that flag missing idempotency. Wire a second model to review the first. *Better answers* is the entire arc of the current research frontier — better prompts, better RAG, better post-training, better tool use. The bet is that if we can get the agent to answer the hard questions correctly, the code will be safe.

The bet has a problem. The number of questions is not fixed. Every new endpoint, every new entity, every new integration introduces fresh questions, and the questions interact with each other. Two endpoints that are each individually correct can interact incorrectly. Three services that each handle retries correctly can produce a retry storm together. The questions don't compose. The answers don't compose. Even if the agent gets ninety-nine percent of the local questions right, the one percent it gets wrong shows up at the seams, where no single endpoint owns the failure. The probability of a correct distributed system is the product of the answers being right, not the average. Better answers improve the average. They do not save the product.

The other bet — Reboot's bet — is that the right move is in the opposite direction. Don't ask better questions. Don't train better answers. *Remove the questions.* If the agent never has to decide what isolation level it's running at, it cannot pick wrong. If the agent never has to write a retry loop, it cannot make the retry loop unbounded. If the agent never has to wire up an idempotency key, it cannot forget the key. The questions that used to live in the application disappear because the runtime answers them once, globally, and never asks the application what it thinks.

What survives, on the application side, is the business logic. *When a user sends money, decrement the sender, increment the recipient.* That sentence is the part the agent should be writing. It contains no question about isolation, retry, recovery, or ordering. Those questions are not absent because the agent is good at avoiding them — they are absent because the runtime took ownership of them at a layer the application doesn't reach into. The agent writes entities and actions. The runtime executes them with exactly-once semantics, durable state, serialized access per entity, and recovery across crashes. The application speaks a smaller language. The smaller language has fewer places to be wrong.

This is structurally different from abstraction. Abstraction hides a question by giving it a default — the application can override the default if it wants to, which means the question still exists, just deferred. The default leaks. The override is dangerous. Every senior engineer has a story about the time the framework's "sensible default" turned out to be wrong for their use case, and they had to drop into the layer below and fight with the framework to do the thing they actually needed. Reboot is not hiding the question with a default. The question is gone. The application cannot ask it because the application does not have the words. There is no isolation-level setting because there is no transaction the application owns. There is no retry config because there are no retries the application controls. The dangerous surface area has been deleted, not abstracted.

This is what *fewer questions, not better answers* means as an engineering bet. It says the path to safe AI-written distributed code does not run through making the agent smarter. It runs through making the surface the agent writes against smaller and stricter. The agent's job is to express the business logic, which is the part it is genuinely good at — straight-line, single-actor, in the small. The runtime's job is everything that requires reasoning about the world being concurrent, unreliable, and partial — the part the agent is structurally bad at and will keep being bad at, because the bad-at-it is downstream of the agent's lack of operational scar tissue, not its lack of training data.

The implication is uncomfortable for anyone betting on "agents will get better at writing distributed code." They will. They will get incrementally better forever and they will never be good enough, because the failure modes that matter are the ones that show up under conditions the agent has never seen and cannot simulate. The only durable answer is to take the failure modes off the table. Not by asking the agent to handle them. By writing the runtime so that the failure modes are not states the system can enter.

Every question removed from the application is a question that cannot be answered wrong. Every answer the runtime owns is an answer that is the same across every endpoint, every team, every model version, every retry. The agent writes less. What it writes is more obviously correct. The system, taken together, behaves like the runtime promised — because the application never had the chance to break the promise.

That is the whole bet. Fewer questions. Not better answers.
