---
title: Framework Comparison
description: >-
  How hiding the hard parts made UI safe to write at scale, and why backends are
  next.
---
Before React, front-end development was a fight with the DOM. You wrote jQuery that selected elements, mutated their attributes, attached handlers, and prayed the state inside your JavaScript variables stayed in sync with the state visible on the page. It didn't, of course — not under any nontrivial app. You'd update a counter in three places because three places displayed it. You'd forget the fourth and ship a bug where one corner of the UI lagged behind the others. You'd attach a click handler, then re-render the parent, then notice the click handler was gone because the element had been replaced. The bugs were endless. The class of bug was structural — anywhere your in-memory state and your rendered DOM could disagree, they eventually did.

React did one thing. It hid the DOM. The developer stopped writing instructions to the DOM and started declaring what the UI should look like, given the current state. The runtime — the React reconciler — figured out the minimum set of DOM operations to make reality match the declaration. The developer never called `appendChild` again. The whole class of state-versus-view bugs disappeared, not because developers got better at remembering to update both places, but because the second place was no longer reachable from the developer's code. The DOM became a thing the runtime owned. The developer wrote components.

A decade later, this looks like a small change. At the time it was structural. It moved an entire category of correctness — *the in-memory state matches the rendered output* — from the application's job to the platform's job. Every UI framework that followed (Vue, Svelte, SwiftUI, Compose) adopted the same move. The DOM-manipulation era ended. It ended because somebody built the platform that made it ungrateful work.

Reboot is the same shape of move, applied to a different platform layer. Replace "DOM" with "distributed state, transactions, ordering, and recovery," and the parallel writes itself. Before Reboot, building distributed backends was a fight with state. You wrote application code that loaded rows, modified them, wrote them back, retried on failure, handled idempotency, journaled to a queue, processed events, recovered from crashes, and reconciled drift. The fight produced the same shape of bug React's fight produced — the application's mental model of the system's state didn't match the system's actual state, and the gap leaked everywhere. You'd debit an account in three places and forget the fourth. You'd write an idempotency key that worked for the obvious retry and missed the subtle one. You'd handle the crash case for ninety percent of operations and ship the tenth as time-bombed. The bug class was structural. Anywhere the application owned a distributed-systems concern, that concern eventually got handled wrong.

Reboot hides the distributed state and the operations on it. The developer stops writing instructions to the database and starts declaring what an entity is and what actions it can take. The runtime — the Reboot runtime — figures out the minimum set of operations needed to make state durable, serialized, ordered, and recoverable. The developer never writes a transaction, never takes a lock, never generates an idempotency key, and never wires a retry. The whole class of distributed-correctness bugs disappears, not because developers got better at remembering to handle the seams, but because the seams are no longer reachable from the developer's code. The transaction layer became a thing the runtime owned. The developer writes entities and actions.

The parallel is exact in shape and important in one difference. React hid the DOM, which is a single, well-defined browser API. Reboot hides distributed-systems infrastructure, which is a sprawling collection of decisions — database isolation, queue ordering, retry policy, lock granularity, idempotency strategy, crash recovery, ordering guarantees, audit logging. There is no single API equivalent to "the DOM." There is a whole stack of conventions that the application code historically had to participate in. Reboot has to absorb that whole stack into a runtime contract. That is more work than React did. The reward, if it works, is bigger — distributed-systems bugs are more expensive than UI bugs, and the labor freed up is higher-leverage.

The skeptic's question is whether this kind of move is actually possible at the backend layer. UI is well-bounded; the DOM has a finite API and a small set of failure modes. Backends touch unbounded external systems, network partitions, hardware failures, and third-party APIs. Can a runtime really hide all of that and still let the application be expressive enough to do useful work?

The same question was asked about React. *Sure, you can hide the DOM for trivial UIs, but real apps need to drop down and manipulate the DOM directly for performance, for integration with non-React widgets, and for animations.* The answer turned out to be no — they didn't. React grew into the high end. Performance got addressed by the reconciler getting smarter. Integration got addressed by component libraries. Animations got addressed by APIs declared in React's vocabulary. The escape hatches were used by a small fraction of users for a small fraction of cases, and the vast majority of UI code shipped without ever calling them.

Reboot's bet is the same. The escape hatches will be used. They will be used rarely. The vast majority of backend code — the CRUD, the workflows, the transactions, the integrations with payment processors and email senders and ledger services — will ship without ever touching the dangerous primitives because the primitives have been folded into the runtime and the runtime makes the obvious version of the code work. The escape hatches will exist for the cases that need them, but they will be the exception. Today, on most stacks, the dangerous primitives are the rule. Every endpoint takes a lock. Every retry has a config. Every transaction is wrapped by hand. The "exception" is most of the file.

What changes when this shift happens? The same thing that changed when React shipped. The shape of the application code gets smaller. The shape of the bug list gets shorter. The shape of the team gets different — fewer engineers writing infrastructure, more engineers writing product. The shape of the agents writing the code gets viable. Front-end work became something you could ship from an idea to a deployed UI without a senior engineer reviewing every DOM call. Backend work, on Reboot, becomes something you can ship from an idea to a deployed system without a senior engineer reviewing every transaction.

The category React occupied — *the substrate that made UI safe to write at scale* — is now a $400B market. The category Reboot occupies — *the substrate that makes distributed backends safe to write at scale* — is bigger because the backend is where the money lives. The pattern is the same. The leverage is the same. The right framework arrives, hides the dangerous primitives, and a generation of developers (and, this time, agents) stops having to know they exist.

Reboot is to AI-written code what React was to UI code. A substrate that hides the hard, error-prone parts behind primitives that can't be misused. The hard parts here are state, transactions, and distributed consistency. The agents — and the humans — never have to touch them again.
