---
title: Historical
description: 'The history of computing is the history of substrates absorbing burdens. '
---
The history of computing is the history of moving hard problems out of the application and into a layer underneath it. Every era has its own version of the move. The applications that came before the move had to do the hard thing themselves, badly and inconsistently. The substrate that absorbed the hard thing made it consistent, correct, and invisible. Within a decade of each move, the previous arrangement looked unimaginable. *You had to do that yourself? In every program?* Yes. We did. Until we didn't.

The first move was scheduling. In the batch era, the application owned the machine. It ran end to end with exclusive access, finished, and yielded the box to the next job. This was fine when a single job took a day and a machine cost a million dollars and you wanted to amortize the box. It stopped being fine when multiple jobs wanted to run at once. The time-sharing operating system absorbed scheduling. Applications stopped owning the CPU and started getting time slices. The kernel decided who ran when. The applications didn't have to know that other applications existed. *Scheduling* moved from a thing programmers had to write into a thing the system carried.

The second move was memory. Early multi-user systems exposed shared memory directly. Applications could read each other's address spaces, scribble on each other's data structures, take down the whole machine by writing past the end of an array. Virtual memory absorbed this. The kernel gave each application its own address space and refused to let one process see another's memory. Applications didn't have to know how to coexist. They just couldn't reach each other. *Memory safety* moved from a discipline programmers had to maintain into a property the platform enforced.

The third move was durability. Early business applications wrote their own file formats, managed their own transactions, and recovered from crashes by hand. Every accounting system had a custom recovery script. Every ledger had its own consistency model. Then the relational database arrived. The application stopped owning the durability story. It declared schemas, ran queries, and trusted the database to commit, replicate, recover, and reconcile. The application didn't have to know how transactions worked. It declared them and the database delivered them. *Durability and consistency* moved from a per-app problem into a substrate guarantee.

The fourth move was rendering and isolation. Documents on the web in 1995 had to be rendered, fetched, and sandboxed by whatever could read them. The browser absorbed all of it. Every webpage gets the same rendering engine, the same network primitives, the same sandboxing. The web app doesn't have to know how to fetch a remote resource, how to parse HTML, how to keep one tab from poisoning another. The browser handles it. The web app declares its DOM and trusts the platform. *Rendering, networking, and isolation* moved from an application problem into a browser guarantee.

The fifth move was provisioning. In 2005, deploying an application meant procuring servers, racking them, installing the OS, configuring the network, monitoring the boxes, handling failures by hand. Virtualization absorbed some of this. Containers absorbed more. Kubernetes absorbed the rest. The application stopped owning the deployment story. It declared a manifest and trusted the orchestrator to schedule it, recover it, scale it. The application didn't have to know how to handle node failures. The orchestrator did. *Deployment and infrastructure* moved from an ops problem into a platform guarantee.

Five moves, five eras, five substrates. The pattern is consistent. Each time the move happens, three things are true at once. First, the hard problem has been a per-application burden for long enough that there is a generation of programmers who know how to do it by hand and consider that knowledge their craft. Second, the cost structure has changed underneath in a way that makes the previous arrangement unsustainable — more concurrent jobs, more users, more data, more deployments. Third, somebody builds the substrate, the substrate works, and the per-application burden disappears. The craftsmen don't lose their jobs — they get redeployed onto the next layer up, where the work is now hands-on and the substrate hasn't formed yet. The cycle repeats.

There is one big hard problem left that is still a per-application burden. Distributed correctness. The application is still responsible for getting transactions right across services, for handling retries correctly, for journaling work that needs to survive a crash, for ordering events that have to land in sequence, for deduplicating operations that might fire twice, for reconciling state that drifted because the network blinked. Every application that touches more than one machine has to do this work itself, badly and inconsistently. There are libraries that help — Temporal, Akka, Erlang, Cloud Tasks — but they sit beside the application. The application can use them carelessly. The application can ignore them. The application still owns the distributed-correctness story.

This is the unfinished move. It is the next substrate in the historical sequence. It has not happened yet because, until recently, the cost structure didn't demand it. Distributed code was expensive to write, and the expense was paid out in senior engineers, and the senior engineers, over a career, accumulated the defensive reflexes to make the libraries work. The system was in a long, ugly equilibrium. Distributed bugs happened constantly, but they happened at a rate that scaled with engineer-hours, and engineer-hours were the bottleneck on shipping, so the bug rate was implicitly capped at a level the operations team could absorb.

The cost structure changed in 2023. Agents started writing code at twenty times the rate of human engineers. The bug rate, which had been implicitly capped by engineer throughput, was now uncapped — code shipped at agent speed, bugs shipped at agent speed, and the operations team did not scale by twenty. The previous arrangement stopped being sustainable. The condition that historically triggers a substrate move — the cost of doing the hard thing per-application has become structurally unaffordable — has arrived for distributed correctness. The substrate must form. The only question is who builds it.

This is the historical place Reboot occupies. Not a new framework. Not a better workflow engine. Not a smarter ORM. The next substrate in the sequence. The layer that absorbs distributed correctness the way the kernel absorbed scheduling, the way virtual memory absorbed memory safety, the way the database absorbed durability, the way the browser absorbed rendering, the way Kubernetes absorbed provisioning. Each previous substrate looked, at the moment it arrived, like an opinionated curiosity. Each one became invisible infrastructure within a decade. Each one made the layer below it stop being a thing applications worried about.

The pattern is so consistent it is almost a law. Hard problem persists as per-application burden. Cost structure shifts and makes the burden unaffordable. Substrate forms. Burden moves into the substrate. A generation of craftsmen gets redeployed onto the next unsolved layer. Repeat.

Distributed correctness is up next. The cost structure has shifted. The substrate is forming. The craftsmen, soon, will be writing the layer above the one Reboot makes invisible — products that ship faster, with fewer engineers, because the dangerous parts are no longer in the file. The history is consistent on this. It is a matter of time, not a matter of doubt.

Reboot is the next move. It arrives exactly when AI agents need it to.
