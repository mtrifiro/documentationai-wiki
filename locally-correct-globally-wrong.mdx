---
title: 'Locally Correct, Globally Wrong'
description: Every function is right. The system is wrong. That's the whole problem.
---
LLMs are good at writing a function. Give an agent a clear specification, a single input type, a single output type, and a problem that fits inside one screen, and the code that comes back will usually do what was asked. The reasoning the agent applies is local — it considers the variables in scope, the libraries available, the obvious cases and the obvious edge cases. The function reads. It computes. It returns. Tested in isolation, it passes. This is the kind of code agents have been trained to produce, and they produce it well. Local correctness is the home court of an LLM.

LLMs are structurally bad at the rest. The rest is the part that makes the function correct *in context* — when two of these functions run concurrently against the same row, when one runs after the other has half-failed, when this function is called by a retry loop that has fired three times because the network was slow. None of that context is on the screen with the function. The agent has no view of it. There is no way to read the global behavior of a system out of any single file, and an LLM operating on a few thousand tokens of source has never had access to the global behavior at all. It has access to the function. It writes the function. The function is locally correct. The system is globally wrong.

Two endpoints in the same service illustrate it. Endpoint A increments a counter. Endpoint B decrements the same counter. The agent writes both. Each is a four-line function. Each is, locally, exactly right — read the counter, change it by one, write it back. Now run them. The traffic pattern is bursty. Some seconds, A and B fire dozens of times within the same millisecond. The interleaved reads and writes lose increments and decrements. The counter drifts. Nothing in either endpoint is wrong. The bug lives in the interleave between two correct endpoints. The agent did its job twice. The system failed once.

Three services compose the failure differently. Service X retries on failure with exponential backoff. Service Y retries on failure with exponential backoff. Service Z retries on failure with exponential backoff. Each retry policy is, locally, exactly right — bounded, jittered, reasonable. Wire them in a chain. X calls Y calls Z. Z has a brief outage. Z's caller, Y, retries. Each retry fans out into another call to Z. Y eventually times out and gets retried by X. Each X retry fans out into more Y retries, which fan out into more Z retries. Z's outage extends because Z is now drowning in retry traffic from its own callers. Each retry policy was locally correct. The system, taken together, produced a retry storm. There is no single function to blame. There is no single line of code to fix. The bug is the composition.

Four workers pulling from the same queue compose it again. Each worker grabs a job, processes it, marks it done. Each worker's code is, locally, exactly right. Two workers happen to grab the same job — say the queue's visibility timeout was set wrong, or a worker took too long and the job was returned to the queue while the original worker was still processing. Both workers process the job. Both mark it done. The downstream effects (an email sent, a charge made, an inventory decrement) happen twice. The code is fine. The contract between the workers and the queue is fine, in isolation. The bug is in the assumption that the contract holds — and the assumption is global, not local, and no worker can verify it from inside the function.

This is the structural asymmetry: **local correctness composes additively, global correctness composes multiplicatively.** Two locally correct functions side by side give you two correct functions — a small win, a slightly bigger system that works. The improvement is linear. Two globally correct compositions, on the other hand, multiply — each one strengthens the property of the system as a whole. But the failure mode multiplies the other direction. A single locally correct function that happens to break the global invariant breaks the *whole system*, not just itself. One missing lock in one endpoint produces lost writes across every endpoint that touches the row. One missing idempotency key in one webhook handler produces duplicates across every consumer that reads the resulting events. Local rightness adds. Global wrongness multiplies. You cannot win the game by being locally right enough times — the multiplicative property of the failures dominates.

An agent that is 99% locally correct produces, in a system with a hundred composition points, a system that has a 37% chance of being globally correct at any moment. That is the math. It does not get better by raising the agent's local accuracy to 99.9% — you get 90% global, which is still not a system you can ship a financial product on. The asymptote of local-correctness-only is "almost right." The product never closes the gap to "actually right" because the structure of distributed systems does not let it.

This is why the strategy of *better prompts, better evals, better fine-tuning, better code review* has a ceiling. The ceiling is the gap between local and global correctness. Every one of those interventions improves local correctness. None of them produces global correctness, because global correctness is not visible inside any single function the agent is writing. You can review every line and approve every line and ship code that is, function-by-function, perfect — and the system will still break, because the bugs live in the composition the reviewer cannot see.

Reboot's move is to put global correctness in the only place it can structurally live: the runtime. The runtime sees the whole composition. It sees that two endpoints touch the same entity and serializes them. It sees that an action is being retried and deduplicates it. It sees that a worker has died mid-job and ensures the job either completes or is retried exactly once. None of these guarantees can be made from inside the function the agent is writing, because none of them are facts about that function — they are facts about the function's relationship to every other function in the system. The runtime is the layer that knows the relationships. The runtime is the layer where the global invariants get enforced.

The agent keeps doing what it is good at — writing the local function, naively, straight-line, the obvious shape. The runtime carries the global invariants the agent cannot see. The two together produce a system that is globally correct, not because the agent was smart enough to see the whole picture, but because the picture was held by something that did.

Locally correct, globally wrong, is the failure mode of every AI-written backend in 2026. It is not solved by making the agent better at the local part. It is solved by making the global part somebody else's job. Reboot is that somebody else.
