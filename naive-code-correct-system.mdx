---
title: 'Naive Code, Correct System'
description: 'The obvious version of the code should be the correct version. '
---
The first version of any function is the naive one. You write what the spec said. You handle the case the user described. You return the value they asked for. The function reads like the sentence the requirement was written in — *take the user's input, validate it, save it, return success*. It is short. It is direct. It is what a careful person would describe out loud if you asked them how the feature should work. It is also, on every framework built in the last forty years, broken in production. The naive version handles the case the requirement described. The requirement described a single user, a working database, a non-flaky network, an unambiguous sequence of events. Production has none of those.

So the careful engineer writes the second version. The second version adds the lock, because two users might hit this at once. The third version adds the retry, because the database connection might drop. The fourth version adds the idempotency key, because the client might fire the request twice. The fifth version wraps the operation in a transaction, because partial writes are not allowed. The sixth version handles the case where the transaction commits but the response never returns. The seventh version writes to a queue instead of directly, because the work needs to survive a crash. By the time the function is production-ready, it bears no resemblance to the naive version. It is fifty lines of defensive code surrounding three lines of business logic. The defensive code is most of the file. The business logic is buried.

This is what frameworks reward. The careful engineer learns, over years of pages and postmortems, which defensive patterns to apply where. The naive version is what beginners write. The defensive version is what seniors write. The path from one to the other is paved with incidents. *Yes, you can write it that way, and it will work for the first six weeks, and then it will break in this specific way, and that's why we do it differently.* Senior engineers earn their salary by being the people who have already paid the tuition. The codebase fills up with their lessons. The naive version stops being a thing anyone ships.

The problem with this pattern is that it scales with the writer. It works when the writer is the careful engineer who has been through the postmortems. It does not work when the writer is somebody else — a junior engineer, a contractor, an agent. The defensive patterns are not visible in the spec. They are not visible in the requirement. They are not visible in the docs. They are the dark matter of the codebase, the unwritten knowledge that senior engineers carry in their heads and apply by reflex. When the writer doesn't have the reflex, the writer writes the naive version. The naive version ships. Six weeks later it breaks.

Agents write the naive version. Always. They were trained on a corpus where most of the code worked, but the corpus does not encode why the code worked — which lock prevented which race, which retry policy bounded which storm, which idempotency strategy caught which duplicate. The agent reads the function. The function reads as naive code that worked. The agent reproduces naive code that won't work, and ships it. This is not a fixable training-data problem. The defensive knowledge is too contextual to encode at the level the agent operates. The agent will keep writing naive code.

The standard response is to try to make the agent careful. Teach it the patterns. Give it a checklist. Have it consult a defensive-coding guide. This is the strategy of *making the writer match the framework*. It is the strategy that worked, expensively, with juniors and bootcamps and ten-year tenure ladders. It does not scale to agents writing at twenty times the velocity of the engineers who would normally apply the patterns. The agent will pattern-match to the wrong defensive idiom. It will apply the right idiom inconsistently. It will write the lock and forget the retry, write the retry and forget the idempotency, write the idempotency key from `uuid4()` inside the handler so each retry gets a fresh one. The agent's defensive code is a worse failure mode than the agent's naive code, because defensive code that's almost right looks like working defensive code right up until the moment it isn't.

Reboot's bet is the opposite. Don't make the writer match the framework. Make the framework match the writer's natural mode. The natural mode is naive code. The naive version of *transfer money* is *decrement sender, increment recipient*. The naive version of *save profile* is *write the new fields to the row*. The naive version of *process webhook* is *read the event, do the thing the event describes*. These are the sentences the requirements were written in. They are what the writer — human or agent — would say out loud if you asked. Reboot is the runtime that makes those sentences correct.

This is what "naive code, correct system" means as a design target. The system as a whole behaves correctly — under concurrency, under failure, under retry, under crash. The code as written is naive. The gap between the two is held by the runtime. The transaction is there, but the application code doesn't declare it. The lock is taken, but the application code doesn't reach for it. The retry is bounded, but the application code doesn't configure it. The idempotency check happens, but the application code doesn't carry the key. Every defensive pattern that used to bury the business logic is now executed by a layer underneath the file, and the file gets to be the naive version that the requirement implied.

The freed-up writer experience matters. The naive version is faster to write, faster to read, faster to review, faster to change when the requirement changes. The defensive version was a cost paid to a runtime that didn't carry its own weight. The runtime that carries its own weight removes the cost, and the writer recovers the time. For a human, the recovered time goes into product thinking, into the parts of engineering that actually move the business. For an agent, the recovered time goes into shipping the next feature, because the agent didn't have time-cost for defensive code anyway — what it had was failure cost, which dropped to zero.

The deeper claim is that the "naive code" was always the right code. The defensive patterns were not virtues. They were workarounds for runtimes that pushed correctness up into the application instead of carrying it themselves. A senior engineer's career-long accumulation of defensive reflexes was, in part, a tax extracted by an industry that didn't have the substrate to make the tax unnecessary. Reboot makes the tax unnecessary. The naive code is the correct code. The careful version was the framework's failure, not the writer's wisdom.

You should be able to write the obvious version of the code and still get a correct system. That is the bet. The runtime carries the weight that frameworks have historically pushed onto the developer — durability, ordering, exactly-once execution, failure recovery. The developer writes the sentence the requirement was written in. The system does what the sentence said.
