---
title: Substrate
description: >-
  Moving distributed correctness from the application layer to the platform
  layer.
---
There's a class of bug that compiles, passes review, deploys cleanly, and breaks six weeks later. Two users hit the same endpoint within the same millisecond. A retry fires twice. A batch job crashes at 3am because the queue was empty for the first time in a year. The code wasn't wrong, exactly. It was almost right.

Almost right is the dominant failure mode of AI-written code. Not syntax errors (those get caught by the compiler). Not logic errors in the small (those get caught by tests). The bugs that survive into production are the ones where each function is locally correct and the system, taken together, is globally wrong. Two endpoints both read the same row, both decide to update it, and both write. One write wins. The losing write disappears. Nobody notices until the customer support ticket arrives, and by then the audit trail is thin and the cause is buried under twelve deploys.

This is not a new class of bug. Distributed systems engineers have been writing about it for forty years. What's new is the rate at which the bug is being introduced. An AI agent can produce a thousand lines of plausible Go in the time it takes a senior engineer to read a hundred. The code looks fine. It passes the obvious tests. It ships. The agent has no model of what 3 am production looks like, no scar tissue from the time the retry storm took down the payments service, and no instinct that says *wait, what happens if this runs twice*. The code is naive in the technical sense; it does not know what it does not know.

The standard response is to ask the agent better questions. Prompt engineering. Specs. Linting. Code review. More tests. More guardrails. More humans in the loop. All of this is downstream of the actual problem, which is that the code is being asked to carry guarantees it has no way to enforce. You can write the most careful prompt in the world, and the agent will still produce a function that reads, computes, and writes. Three operations, three places where the world can change underneath you, three places where correctness is somebody's responsibility and that somebody is the code. The code is the wrong place for that responsibility. The code has always been the wrong place for that responsibility. We accepted it because, until recently, the people writing the code were skilled enough to almost carry it.

The substrate is the layer underneath the code where the guarantees actually live. Not a library you import. Not a framework you opt into. A runtime that the code runs on, where the rules of the universe are different: where two endpoints cannot both win a race because the runtime serializes them, where a retry cannot fire twice because the runtime deduplicates it, where a write cannot disappear because the runtime journals it before the function returns. The code stays naive. The system becomes correct. Not because the code got better, but because the substrate refuses to let the code be wrong about the things it was always going to be wrong about.

This is the move every successful platform layer has made. The operating system absorbed memory and scheduling because asking every program to manage them was a losing fight. The relational database absorbed durability and consistency because asking every application to implement them was a losing fight. The browser absorbed rendering and sandboxing because asking every website to be a good citizen was a losing fight. In each case, the pattern is the same: a class of correctness concern that used to be the application's job becomes the platform's job, and the application gets to be simpler in exchange for giving up the ability to do the dangerous thing at all. You cannot scribble on another process's memory in a modern OS, no matter how much you want to. The substrate forbids it.

Distributed correctness has never had its substrate. It has had libraries — Akka, Temporal, Erlang, Cloud Tasks — that help if you use them carefully and hurt if you don't. They sit beside the application. The application can ignore them. The application can call them wrong. The application is still responsible for understanding the failure modes, holding the locks, retrying in the right order, and deduplicating on the right key. The library is a tool. The substrate is the floor.

Reboot is the substrate move applied to distributed code. The runtime owns state. It owns concurrency. It owns failure recovery. It owns the question of what happens when a write half-completes and the process dies. The application doesn't get to answer those questions because the application doesn't get to ask them. It writes entities and actions — nouns and verbs — and the runtime handles the part where the nouns persist correctly across crashes and the verbs execute exactly once. The naive version of the code is the correct version of the code because the runtime forbids the incorrect version from existing.

This matters more in the agent era than it did before. When humans wrote the code, the substrate gap was filled by experienced engineers who had been burned enough times to write defensively, to hold the right lock, and to bound the retry. The skill was scarce and expensive but it was real. Agents don't have it. Agents have never been burned. Agents will write the obvious version of the code, ship it, and the obvious version of the code is the version where the race condition exists. You can train the agent to write defensive code. You will be training it for the next twenty years, and it will still get it wrong in the cases it hasn't seen. The agent is bad at the same things every junior engineer is bad at, except faster and at scale.

The alternative (better prompts, better reviews, better evals) assumes that the problem is the quality of the code being generated. It isn't. The problem is the location of the correctness guarantees. As long as those guarantees live in the application layer, somebody has to put them there, and the somebody is either an expert who is increasingly out of the loop or an agent who doesn't know what the loop is. Move the guarantees down. Put them in the substrate. The application can be written by anyone — or anything — and the system stays safe.

This is what "safe by construction" means. Not safe because the code was reviewed. Not safe because the tests passed. Safe because the substrate would not let the unsafe version compile and run in the first place. The unsafe version is unreachable. It is not a state the system can enter. Whatever the agent writes, whatever the prompt was, whatever the model's training distribution looked like — the substrate holds the line.

The bet is that this is where the value of the stack moves next. Agents will keep writing code. They will keep writing it faster. They will keep writing it almost right. The layer that turns almost right into actually right — that's the substrate. That's Reboot.
