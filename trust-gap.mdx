---
title: Trust Gap
description: >-
  Why the gap between shipping and correctness can no longer be bridged by
  humans.
---
There is a gap between the moment a deploy turns green and the moment the system is actually working. Most of the time the gap is invisible — the code does what it looks like it does, the traffic stays inside the shape the tests anticipated, the database doesn't blink. The gap is still there. It widens at night, under load, during the deploy you didn't notice was overlapping with the migration. Eventually something falls into it. A charge runs twice. An inventory count goes negative. A user sees somebody else's data for forty seconds. The gap had been open the whole time. The deploy was green because nothing had stress-tested it yet.

This is the trust gap. The distance between code that compiles and code that actually works at 3am under load. It has always been the real surface area of software engineering — not the part you can see in the diff, but the part you have to predict from the diff. Every line of code is a small promise about how it will behave under conditions that aren't visible at the moment it was written. Concurrent calls. Partial failures. Retries. Network partitions. Disk full. Clock skew. The line of code doesn't mention any of these. It runs in a universe where they don't exist. The runtime puts it in a universe where they all do. The gap is everything the code didn't say but is responsible for anyway.

Software has lived inside the trust gap for as long as software has been distributed. The way we coped with it was social. The senior engineer reviewed the PR and said *what happens if this gets called twice*. The on-call engineer paged at 4am and learned, the hard way, that the retry was unbounded. The architecture review caught the race condition before it shipped. Trust was patched in by people. Every team had a few engineers who had been burned enough times to instinctively close the gap, and the rest of the team trusted those people to catch the things the code couldn't say. The trust gap was real, but it was bridged by scar tissue. It worked, expensively and unevenly, for a long time.

It doesn't work anymore. The agent era breaks the bridge in two places at once.

First, the volume of code is wrong. An agent can write a thousand-line service in an hour. The senior engineer who used to catch the race condition cannot read a thousand lines an hour and catch the race condition. Even if they could, there are now twenty agents writing in parallel. The review queue is no longer a queue — it is a fire hose, and the fire hose is producing code at a rate that makes serious review structurally impossible. The team's most experienced engineers, the ones who used to be the trust patch, become the bottleneck. Either they slow down the agent — which gives back the speed advantage — or they wave it through, and the trust gap opens wider than it ever was.

Second, the code itself is differently wrong. Human engineers write bugs in patterns. They forget the same locks, they misorder the same retries, they introduce the same off-by-ones. After a decade in production, a senior engineer has a mental model of where bugs live and what they smell like. Agents write bugs in different patterns — confident bugs, plausible bugs, bugs that look exactly like the working code next to them because the agent has no internal sense of which lines are load-bearing. The reviewer's instincts misfire. The diff looks fine. It is, locally, fine. It will not be fine in production.

The standard response to this is to harden the social patch. Better prompts. Better specs. Better evals. More tests. AI reviewers reviewing AI code. All of these treat the trust gap as a quality-of-code problem — if the code were better, the gap would close. The trust gap is not a quality-of-code problem. The trust gap is a *location-of-guarantees* problem. As long as the guarantees live in the code — as long as correctness under concurrency, failure, and retry depends on the application doing the right thing at every call site — the gap will exist. Better code makes the gap narrower. It does not close it. The only way to close the gap is to move the guarantees out of the code entirely.

This is what Reboot does. The runtime, not the application, is responsible for the parts of correctness that the code was never going to be reliably right about. Concurrent writes to the same entity are serialized by the runtime, not by a lock somebody remembered to take. Retries are deduplicated by the runtime, not by an idempotency key somebody remembered to generate. Partial failures are recovered by the runtime, not by a compensating transaction somebody remembered to wire up. The application says *charge this card and update this balance*. The runtime is the thing that guarantees the charge and the update happen together, or neither happens, no matter how many times the network hiccups or the process dies or the same request gets retried by an over-eager client. The application's promise is small and local. The runtime's promise is large and global. The trust gap closes because there is no longer a place for it to open.

The shift is the same one operating systems made for memory. In the early days of computing, every application was responsible for not scribbling on another program's memory. They scribbled on each other's memory constantly. The fix wasn't to write better applications — it was to put the memory manager in the kernel and make scribbling structurally impossible. The trust gap around memory closed not because programmers got more careful, but because the place where trust was required moved from the application to the platform. The application got to be simpler. The platform got harder. The system, taken together, got safer than any pile of careful applications could have been.

Distributed correctness is the next place this move has to happen, and it has to happen now, because the rate at which untrusted code is being generated has gone vertical. You cannot patch the trust gap with experience anymore. There isn't enough experience to go around, and the code is being written faster than experience can be applied to it. The only durable answer is to stop trusting the code at all — to design the runtime so that whether the code is correct or almost right or outright wrong about the distributed-systems parts, the system behaves correctly anyway.

This is what *safe by construction* means in practice. Not safe because somebody checked. Safe because the unsafe behavior is not a state the runtime will allow the system to enter. The agent writes naive code. The runtime makes the system correct. The trust gap that used to be patched by humans, then by tests, then by reviewers, then by anxious 4am pages, becomes a gap that doesn't exist — not because we got better at watching it, but because we moved the responsibility for closing it to the layer that can actually keep it closed.

That layer is Reboot. The trust gap is what it's for.
